    Lifecycle¶

    This section describes, step by step, exactly what Vibrator does at the two moments that matter most:

    • What happens on build

      The deterministic five-stage Dockerfile generator — every layer, in order, from the Ubuntu base to the final ENTRYPOINT/CMD.

    • What happens on start

      The docker run invocation, then the entrypoint and the per-session claude-exec wrapper — config merging, rules, settings, MCP wiring, and credential plumbing.

    • Runtime detection

      How vibrate finds your Docker socket across Docker Desktop, OrbStack, Colima, Rancher Desktop, Podman, and native Linux.

    The big picture¶

    flowchart TD
    subgraph resolve [Resolve]
      A[Load .vb + flags] --> B[Wizard if needed]
      B --> C[Resolve features + extensions]
      C --> D[Compute fingerprint → image tag + container name]
    end
    subgraph decide [Decide]
      D --> E{Container status?}
    end
    E -- running --> X1[docker exec]
    E -- exited/created/dead --> X2[docker start → exec]
    E -- none, image present --> X3[docker run]
    E -- none, no image --> BUILD[docker build] --> X3
    X1 & X2 & X3 --> ENTRY[Entrypoint + claude-exec] --> AGENT([Agent / shell])

    The Resolve and Decide phases live in the orchestrator (internal/app). The build happens only when no matching image exists (or you pass --rebuild). The startup wiring runs inside the container on every entry.

    Where the decision is made¶

    vibrate inspects the container by its computed name and branches:

    Container state What vibrate does
    Running docker exec straight in
    Exited / created / dead docker start, then docker exec
    Missing, image exists docker run (skips build)
    Missing, no image docker build, then docker run
    --rebuild passed Remove any container, rebuild --no-cache, run fresh

    This is the entire reason the first run in a workspace is slow and every run after is instant: the expensive docker build only happens when the resolved variant has no image yet.
