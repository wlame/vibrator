    CLI reference¶

    vibrate (alias vb) is built with Cobra. Every subcommand lives on its own page below, with all of its flags, defaults, and behavior.

    vibrate [command] [flags]

    Running vibrate with no subcommand behaves exactly like vibrate run. Global flags --help / -h and --version are available everywhere.

    All commands¶

    Command Purpose Reference
    vibrate (bare) Resolve .vb + flags, run the wizard, build/run/exec launch.md
    vibrate run Explicit form of the bare invocation launch.md
    vibrate shell Same as run, but launch your shell instead of the agent launch.md
    vibrate build Build the image without running a container build.md
    vibrate build-dockerfile Emit the generated Dockerfile to a file / stdout build.md
    vibrate wizard Run the setup wizard standalone (no build, no save) wizard.md
    vibrate reconfigure Re-run the wizard and rebuild, preserving credentials wizard.md
    vibrate update Update the harness's CLI in place (no full rebuild) update.md
    vibrate variants list List managed images + containers for this host variants.md
    vibrate variants prune Remove stopped containers / unused images variants.md
    vibrate extensions list List extension entries for a harness extensions.md
    vibrate extensions show Show an extension entry's frontmatter + body extensions.md
    vibrate integrations Interactive setup for host-side integrations integrations.md
    vibrate prereqs status Probe host prerequisites (e.g. claude-mem server) prereqs.md
    vibrate prereqs bootstrap Run host-side setup (e.g. mint a claude-mem key) prereqs.md
    vibrate hostprobe Show host-detected plugins + extension matches hostprobe.md
    vibrate runtime detect Show the auto-detected Docker socket runtime.md

    Spec-resolution flags¶

    These flags shape what gets built. They are shared by vibrate, run, shell, build, and build-dockerfile. The same values can be pinned in .vb; flags always override the pin.

    Flag Type Default Description
    --harness=<id> string — (required) claude-code, codex, opencode, pi
    --profile=<id> string full minimal, backend, frontend, full
    --shell=<id> string zsh bash, zsh, fish
    --with=<feat,...> list — Features to enable on top of the profile
    --no=<feat,...> list — Features to disable
    --extensions=<id,...> list — Extension IDs to install
    --username=<name> string host user Unprivileged user created in the container
    --host-uid=<n> int host UID UID baked into the image
    --host-gid=<n> int host GID GID baked into the image

    Resolution order

    For every field: CLI flag → .vb value → built-in default. The full decision tree is documented in Lifecycle.
